## Problem Statement

Build a local AI-driven trading research tool that runs autonomously every morning, analyzing US stocks through a multi-agent debate system (bull vs bear analysts, risk management, trader synthesis), and producing actionable signals with precise entry/exit/stop levels. The tool runs on a PC with an RTX 4090 GPU, combines two open source repos (daily_stock_analysis + TradingAgents), and keeps cloud API costs under $100/month by using local LLMs for routine tasks.

## What Makes This Cool

Two things. First, signal accuracy from a multi-agent debate — not one LLM guessing, but specialized agents (fundamentals, technicals, sentiment, news) feeding into bull/bear researchers who argue, with a trader synthesizing and risk management approving/rejecting. Second, watching that debate unfold — seeing the bull case, the bear case, the risk assessment, and the final decision with reasoning. It's a hedge fund investment committee running on your laptop.

## Constraints

- **Hardware:** Single RTX 4090 (24GB VRAM). Must run all local inference within this budget.
- **Budget:** <$100/month for cloud API fees (Claude, DeepSeek, or similar).
- **Markets:** US stocks only (NYSE, NASDAQ). No international data sources needed.
- **Execution:** Research tool only — generates signals for manual trading. No auto-execution.
- **Data freshness:** Morning routine. Analysis must complete before market open (9:30 AM ET). Estimated runtime: data fetch ~5-10 min, local analysts ~15-20 min, cloud debate/trader/risk ~30-40 min. Total ~60 min. Scheduler at 6:30 AM gives a 3-hour buffer.
- **Privacy:** All data and models run locally. No sensitive portfolio data sent to cloud (only stock analysis queries to cloud LLMs).
- **Resilience:** Fail-open on data. If a data source times out (YFinance outage, Alpha Vantage rate limit), use stale data with a staleness flag. If a ticker's data fails entirely, skip it and analyze the remaining universe. Log failures, never block the pipeline.

## Premises

1. Combine TradingAgents' multi-agent architecture (LangGraph) with daily_stock_analysis' data pipeline and dashboard patterns into a single system.
2. Local LLM (Qwen 3 MoE or similar via Ollama) handles routine tasks (data summarization, news parsing, technical indicator calculation context). Cloud API (Claude/DeepSeek) handles reasoning-heavy tasks (bull/bear debate, risk assessment, final signal generation). <$100/mo budget.
3. US stocks only. Primary data: YFinance (free) + Alpha Vantage + Alpaca (market data + paper trading + eventual live execution).
4. This is a research/analysis tool — generates signals and analysis for manual trading. No auto-execution, no broker integration yet (Alpaca paper trading is the exception).
5. Runs locally on PC (4090 GPU) as a morning routine — not deployed to cloud, not a web service for others.
6. Paper trading via Alpaca + backtesting from day one, so signal accuracy can be validated before trusting with real decisions.

## Cross-Model Perspective

Codex (independent cold read) proposed a **self-grading investment committee** — agents publish claims with probabilities, and after market close the system scores which agent was right and reweights for tomorrow. This is the ideal architecture (Approach B) that we'll build toward after the vertical slice ships.

Codex identified "watching the agent debate unfold" as the emotional signal — the thing that will make this tool worth opening every morning.

## Approaches Considered

### Approach A: Vertical Slice First (CHOSEN)

Fork TradingAgents, strip to US-only, add YFinance+Alpaca data layer, run 7 agents on 12 tickers, output strict JSON signals. One FastAPI page with debate transcript. Morning cron job. SQLite storage.

- **Effort:** M (CC: ~2-3 hours to scaffold, 1-2 days to integrate, ~1 week for prompt tuning and validation)
- **Risk:** Low
- **Pros:** Ships fastest. Validates core thesis. Strict JSON schema enables scoring later.
- **Cons:** No self-grading loop yet. Limited universe. Minimal UI.

### Approach B: Self-Grading Committee

Full system with morning analysis + after-close scoring. Agents produce claims with probabilities. System tracks hit rates per agent and regime. Reweights agent influence over time. Tiered model routing. Alpaca paper trading. Replayable debate UI.

- **Effort:** L (CC: ~1 week, ongoing tuning)
- **Risk:** Medium — scoring/reweighting is novel
- **Pros:** Signal accuracy improves over time. Compelling debate replay. Paper trading validates.
- **Cons:** More moving parts. Reweighting overfitting risk. Takes longer to ship.

### Approach C: Data Pipeline First

Start with deterministic data pipeline and backtesting infrastructure. Validate agents on historical data before going live.

- **Effort:** M-L (CC: ~3-4 days)
- **Risk:** Low
- **Pros:** Evidence-based validation. Reusable data layer.
- **Cons:** Delayed gratification. Expensive backtesting. Less exciting.

## Recommended Approach

**Approach A: Vertical Slice First.** Get real signals flowing in days, not weeks. The strict JSON schema from day one is the foundation that makes the self-grading system (Approach B) possible in week 2+.

### Model Routing Strategy

| Task | Model | Location | Est. Cost |
|------|-------|----------|-----------|
| Technical analysis | Qwen3-30B-A3B (Q4) | Local (Ollama) | $0 |

| Risk assessment | DeepSeek-V3 | Cloud API | ~$0.15/run |

**Cost scenarios** (12 tickers, assuming ~2k tokens input + 1k output per agent call per ticker):

| Scenario | Daily | Monthly |
|----------|-------|---------|
| Cloud-only (all agents on cloud) | ~$25-30/day | ~$100+/mo |
| **Target hybrid (analysts local, debate/trader/risk cloud)** | **~$15-20/day** | **~$60-80/mo** |
| Aggressive local (only trader on cloud) | ~$5-8/day | ~$20-30/mo |

Note: Start with cloud for everything to validate quality, then progressively move analyst tasks to local once you confirm Qwen 3 quality is sufficient. If cloud-only blows the budget in week 1, migrate analysts immediately.

### Recommended Local Models for RTX 4090

| Model | Params | VRAM (Q4_K_M) | Speed | Best For |
|-------|--------|---------------|-------|----------|
| **Qwen3-30B-A3B** | 30B (3B active) | ~8GB | ~80-120 tok/s (conservative; benchmarks vary) | Analysis grunt work. Fast, cheap, good quality. |
| **Qwen2.5-32B** | 32B | ~22GB | ~30 tok/s | Higher quality analysis when speed isn't critical |
| **Llama 3.3 8B** | 8B | ~6GB | ~100 tok/s | Quick summarization, data extraction |
| **DeepSeek-V2-Lite** | 16B MoE | ~10GB | ~80 tok/s | Alternative to Qwen for analysis diversity |

**Recommendation:** Start with Qwen3-30B-A3B as the default local model. It fits easily in 24GB with room for KV cache and handles structured financial analysis well. Benchmark on YOUR hardware before committing — MoE inference speed varies by framework and quantization.

### Key Technology Decisions

1. **LangGraph for agent orchestration** — inherited from TradingAgents. Structured state management, deterministic flow control, explicit stage transitions.
2. **SQLite for storage (WAL mode)** — simple, local, no server needed. Stores signals, debate transcripts, price data, scores. WAL mode enables concurrent reads during writes. Process tickers sequentially in v1 to avoid write contention. Switch to Parquet for historical price data if volume warrants it.
3. **FastAPI for dashboard** — already in both source repos. Single page, no React overhead for v1. Server-side rendered HTML with `<details>`/`<summary>` tags for expandable debate sections (no JavaScript required). Main view: table of [Ticker, Decision, Confidence, Entry Zone, Target, R/R]. Click row to expand inline bull/bear/risk sections.
4. **Ollama for local inference** — standard interface, easy model switching, well-supported on macOS/Linux with GPU.
5. **LiteLLM for cloud routing** — inherited from daily_stock_analysis. Unified interface across Claude, DeepSeek, OpenAI. Makes model swapping trivial.
6. **Alpaca SDK for market data + paper trading** — single integration for data, paper trades, and eventually real execution.

## Open Questions

1. **Debate rounds:** v1 uses a fixed 1-round debate (bull states case, bear states case, no rebuttals). v1.1+ adds multi-round with rebuttals. If v1 signals feel shallow, the fix is better prompts, not more rounds.
2. **Cloud model choice:** Claude Sonnet 4.5 vs DeepSeek-V3 for the reasoning layer. DeepSeek is ~5x cheaper but Claude may produce sharper analysis. Test both and compare.
3. **Position sizing:** Should the risk manager suggest position sizes (as % of portfolio), or just approve/reject? Include it — even a simple Kelly-criterion-based suggestion is useful.

### Resolved for v1

- **Signal frequency:** Every ticker gets a full scan daily. Simpler, avoids conditional logic complexity. Optimize later if costs are too high.
- **Alpaca paper trading:** Deferred to v1.1. v1 focuses on signal generation and debate quality. No `trading/` module in v1.
- **Ticker processing:** Sequential (one at a time) to avoid SQLite write contention and simplify debugging.

## Next Steps

1. **Clone and explore TradingAgents** — understand the LangGraph state machine, agent prompts, and data flow. This is your foundation.
2. **Set up local Ollama with Qwen3-30B-A3B** — verify it runs on your 4090, benchmark speed, test with sample financial analysis prompts.
3. **Build the data pipeline** — YFinance for price/technicals, Alpha Vantage for fundamentals/news. Store in SQLite. Get 12 tickers loading cleanly.
4. **Wire up the agent pipeline** — adapt TradingAgents' graph to your data pipeline and model routing. Local models for analysts, cloud for debate/trader/risk.
5. **Define the signal schema** — Pydantic model with strict types. This is the contract between agents and storage/display.
6. **Build the FastAPI dashboard** — one page, today's signals, expandable debate transcripts.
7. **Set up the scheduler** — cron with `TZ=America/New_York` at 6:30 AM. Add failure notification (desktop alert or log file).
8. **Prompt tuning** — allocate 4-6 hours per agent. Use few-shot examples from real market scenarios. Test on 3-5 historical setups (pick dates where you know what the right call was) before going live.
9. **Run for 1 week, review signals manually** — are the debates substantive? Are the entry/stop/target levels reasonable? Iterate prompts based on real output.
